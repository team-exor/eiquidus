# Bug Report: Undeterministic Parameter Ordering in RPC Calls

**Commit:** 671c27dfa20ac16c595cdde9e4170a825d7e12f2  
**Date:** January 18, 2026  
**Severity:** Medium (intermittent but causes API failures)  
**Status:** Fixed

## Description

The eiquidus explorer's `/api/getrawtransaction` endpoint occasionally fails with a type error from the Bitcoin/Blackcoin daemon when query parameters are passed in non-alphabetical order. The daemon rejects the RPC call with:

```
Error: Wrong type passed:
{
    "Position 1 (txid)": "JSON value of type number is not of expected type string"
}
```

## Root Cause

In `lib/nodeapi.js`, the `getrawtransaction` and `verifymessage` API endpoints used `Object.keys(query_parameters).forEach()` to iterate over query parameters. This caused parameters to be added to the RPC call in random order:

```javascript
case 'getrawtransaction':
  Object.keys(query_parameters).forEach(function(parameter) {
    if (parameter == 'txid')
      params.push(query_parameters[parameter]);
    else if (parameter == 'decrypt')
      params.push(parseInt(query_parameters[parameter]));
  });
```

### Why This Failed

1. **`Object.keys()` doesn't guarantee order** - While modern JavaScript engines often maintain insertion order, this is not guaranteed across all environments and implementations
2. **HTTP query strings are unordered** - According to RFC 3986, `?txid=abc&decrypt=1` is semantically identical to `?decrypt=1&txid=abc`
3. **Proxies/CDNs may reorder** - Many reverse proxies alphabetically sort query parameters for caching efficiency
4. **Express parsing varies** - The order Express adds keys to `req.query` depends on how the URL string was received

When parameters were processed as `decrypt` before `txid`, the daemon received:
```javascript
getrawtransaction(1, "abc123...")  // Wrong!
```

But it expected:
```javascript
getrawtransaction("abc123...", 1)  // Correct
```

The daemon's parameter validation caught this and rejected the call.

## Why the Bug Was Intermittent

The bug only manifested when query parameters arrived in non-alphabetical order:

### Scenario A: Alphabetical Order (Always Fails)
```
URL:  /api/getrawtransaction?decrypt=1&txid=abc123
Parsed: {decrypt: "1", txid: "abc123"}
Keys:  ["decrypt", "txid"]
Params: [1, "abc123"]  ← ERROR: Position 1 is number, should be string
```

### Scenario B: Request Order (Works)
```
URL:  /api/getrawtransaction?txid=abc123&decrypt=1
Parsed: {txid: "abc123", decrypt: "1"}
Keys:  ["txid", "decrypt"]
Params: ["abc123", 1]  ← SUCCESS: Correct order
```

### Scenario C: Proxy/CDN Reordering
If a proxy or CDN alphabetically sorts parameters, requests that work in testing fail in production, or vice versa. This explains why the bug appeared suddenly or intermittently.

## Solution

Changed from unordered iteration to **explicit parameter handling in correct order**:

### Before:
```javascript
case 'getrawtransaction':
  Object.keys(query_parameters).forEach(function(parameter) {
    if (parameter == 'txid')
      params.push(query_parameters[parameter]);
    else if (parameter == 'decrypt')
      params.push(parseInt(query_parameters[parameter]));
  });

  var cmd = prepareRpcCommand(settings.api_cmds.getrawtransaction, params);
```

### After:
```javascript
case 'getrawtransaction':
  if (query_parameters.txid != null)
    params.push(query_parameters.txid);        // Always first
  if (query_parameters.decrypt != null)
    params.push(parseInt(query_parameters.decrypt)); // Always second

  var cmd = prepareRpcCommand(settings.api_cmds.getrawtransaction, params);
```

Also fixed `verifymessage` case with the same pattern:

```javascript
case 'verifymessage':
  if (query_parameters.address != null)
    params.push(query_parameters.address);
  if (query_parameters.signature != null) {
    let param = decodeURIComponent(query_parameters.signature);
    while (param.indexOf(' ') > -1)
      param = param.replace(' ', '+');
    params.push(param);
  }
  if (query_parameters.message != null)
    params.push(query_parameters.message);
```

## Why This Fix Works

1. **Deterministic order:** Explicit `if` statements guarantee parameters are added in the correct sequence
2. **Null safety:** Checking `!= null` prevents errors when optional parameters are missing
3. **Minimal change:** Preserves all existing logic while fixing the ordering issue
4. **Performance:** Same O(n) complexity for checking parameters, just deterministic
5. **Works regardless of input order:** No matter how the URL is constructed, how the browser sends it, how proxies reorder it, or how Express parses it, parameters are always sent to the daemon in the correct order

## Affected Endpoints

- `/api/getrawtransaction` - Fixed
- `/api/verifymessage` - Fixed

## Affected Coins

**This affects ALL coins using Bitcoin-compatible JSON-RPC.**

The issue is in **eiquidus code**, not the blockchain. Any coin using this explorer that:
- Has `getrawtransaction` or `verifymessage` API enabled
- Receives query parameters in varying order in URLs
- Has an RPC daemon that validates parameter types

Will experience this bug. The error message comes from the Bitcoin/Blackcoin daemon's parameter validation, but the root cause is in eiquidus's `lib/nodeapi.js`.
